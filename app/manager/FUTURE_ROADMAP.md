## 1. Forecasting engine 🟠 Phase 6+

**Goal:** Predict next-month spend and eventually budget, reimbursement, and vendor trends.

**Use cases:**
- Finance: “What will travel spend look like next month?”
- Manager: proactive budget conversations before overrun
- Copilot tool: `analytics.forecast_spend.v1`

**Proposed design:**

```
Historical expenses/claims (12–24 months)
  → seasonal decomposition + rolling averages (MVP)
  → optional Prophet / lightweight ML per tenant (Phase 6+)
  → ForecastResult(month, category, predicted, confidence_band)
```

| Component | Notes |
|-----------|--------|
| `SpendForecastingEngine` | `forecast_month(tenant, department?, category?) → ForecastResult` |
| Data sources | `Expense` (approved), `Claim` (reimbursed), department rollups |
| Outputs | `predicted_spend`, `lower_bound`, `upper_bound`, `drivers[]` |
| Storage (future) | `spend_forecasts(tenant_id, period, category, model_version, payload)` |

**Config:** `FORECASTING_ENABLED`, `FORECAST_LOOKBACK_MONTHS`

**Build order:** team spend forecast → category drill-down → vendor trends → reimbursement pipeline

## 2. Policy impact analytics 🟡

**Goal:** Identify which policies drive the most escalations, rejections, and manager friction.

**Use cases:**
- Policy design: tighten/loosen rules based on escalation rate
- Finance: “Which policy causes the most finance escalations?”

**Proposed design:**

| Metric | Source |
|--------|--------|
| Escalations per policy | `approval_escalations` JOIN `claims.policy_id` |
| Rejection rate | `claims` WHERE `status=rejected` GROUP BY `policy_id` |
| Avg approval delay | `claim_approvals` JOIN `claims` |
| Override rate | `ManagerMemoryService` + high-risk approvals |

**API (future):** `GET /manager/analytics/policy-impact`, tool `analytics.policy_impact.v1`

**Current stub:** `PolicyImpactAnalyticsService.summarize()` returns empty until aggregation jobs exist.

## 3. Manager behavioral risk 🟠 Future advanced

**Goal:** Detect unusual approval patterns for audit and fraud prevention.

**Signals:**

| Signal | Detection |
|--------|-----------|
| Unusually fast approvals | `actioned_at - assigned_at` < P5 threshold (e.g. 2 min) on high-value |
| Repeated overrides | `ManagerMemoryService` stats: `overrides` / `high_risk_approved` ratio |
| Selective policy bypass | Approve high `risk_score` claims repeatedly while rejecting low-risk |
| Batch approve without review | Many bulk approvals in short window |

**Output:** `BehavioralRiskAssessment(score, flags, explanations)` — never auto-block; flag for audit.

**Privacy:** Manager-scoped; visible to `finance_admin` / audit roles only.

**Integration:** Phase 6 dashboard + optional `escalation.create` suggestion (not automatic).

## 4. Export signatures 🟡

**Goal:** Tamper-evident metadata on finance/manager exports for compliance and audit integrity.

**Use cases:**
- Prove export was generated by system at time T
- Detect CSV/HTML modification after download

**Proposed design:**

```
manifest.json + content files
  → SHA-256(content) + HMAC-SHA256(manifest, EXPORT_SIGNING_SECRET)
  → signature block in manifest: { algorithm, content_hash, signature, exported_at, exported_by }
```

| Field | Purpose |
|-------|---------|
| `export_id` | Unique batch |
| `content_sha256` | Hash of CSV/HTML bytes |
| `signature` | HMAC over canonical manifest JSON |
| `exported_by` | user_id, role, tenant_id |

**Verification (future):** `POST /manager/exports/verify` with file upload

**Current:** `ExportSignatureService` attaches placeholder signature when `EXPORT_SIGNATURES_ENABLED=true`.

## 5. SLA breach prediction 🟠

**Goal:** Predict which pending claims are likely to breach approval SLA before they do.

**Signals (combine with `ApprovalPrioritizer`):**

| Signal | Weight |
|--------|--------|
| Hours waiting vs `MANAGER_APPROVAL_URGENT_HOURS` | High |
| Approver backlog size | Medium |
| Historical approver delay (from `ManagerWorkloadAnalyticsService`) | Medium |
| Claim value / employee waiting on reimbursement | Medium |

**Output:** `SLABreachPrediction(claim_id, probability, expected_breach_at, reasons)`

**API (future):** `GET /manager/approvals/sla-at-risk`, tool `analytics.sla_at_risk.v1`

**Actions (future):** notify manager, auto-escalate at 90% SLA consumption (configurable, never default-on)
